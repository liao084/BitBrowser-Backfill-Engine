# RPA 弹窗层级与强制点击问题复盘

> 复盘日期：2026-06-29  
> 对应脚本：`backfill_engine.py`  
> 当前状态：修改后已开始实际运行，当日观察效果良好；长期稳定性仍需持续验证。

> 后续演进（2026-07-16）：本文主体保留弹窗分层修复当时的背景。当前版本已不再以三级弹窗是否存在判定成功；120 秒无心跳后会恢复一级弹窗并重新请求后端缺失量，只有确认无缺失才写入成功结果。

## 1. 背景和设计目标

`backfill_engine.py` 是一个基于 Playwright Async 和比特浏览器 CDP 的历史数据补采 RPA 调度引擎。

脚本的核心设计目标不是“每个日期区块都必须在本轮成功”，而是：

- 一个区块失败不能阻塞后续日期。
- 对高频、可恢复的 UI 问题做局部重试。
- 对无法恢复的当前区块记录原因后跳过，继续遍历总任务期限。
- 保持一个可持续扩展的框架，根据实际运行中出现的问题逐步完善。

本次修改保留了这个 fail-forward（失败后继续推进）策略，没有增加总体任务重试或复杂的断点恢复系统。

## 2. 问题现象

长时间运行后，脚本会偶发出现以下现象：

1. 页面中的“队列任务数”不断减一，最后出现负数。
2. Playwright 日志报告 `span.lostDataBtn` 已经可见、启用且位置稳定，但无法点击。
3. 实际拦截鼠标事件的元素是：

   ```html
   <div class="el-input-group__append">队列任务数(条)</div>
   ```

4. 日志中还出现过“二级弹窗已渲染”，但 `#loseDays_shop_btn` 仍然是 hidden 的矛盾状态。
5. “全店补齐”未成功提交时，旧代码仍会进入 120 秒心跳等待，并可能因三级弹窗不存在而误记为正常完成。

## 3. 三层弹窗的真实 DOM 关系

通过 Chrome DevTools 对 DOM 进行了实际检查。

### 3.1 一级弹窗

一级弹窗是任务检测界面，内部包含：

- 日期范围。
- `#checkbutn` 启动检测按钮。
- `span.lostDataBtn` 补齐数据按钮。
- 缺失数据统计和检测结果。

它的根组件同样是：

```css
div.el-drawer.rtl
```

它的真正唯一特征不是 Drawer 类名，而是内部包含 `#checkbutn`。

### 3.2 二级弹窗

二级弹窗是“缺失数据处理”界面，内部包含：

- 队列间隔时间。
- 队列任务数。
- `#loseDays_shop_btn` 全店补齐按钮。

它的根组件也是：

```css
div.el-drawer.rtl
```

它的唯一特征是内部包含 `#loseDays_shop_btn`。

二级弹窗关闭后不会从 DOM 中删除，而是保留为零尺寸容器：

```text
display: flex
visibility: visible
width: 0
height: 0
```

按照 Playwright 的判定规则，零尺寸元素属于 hidden。

### 3.3 三级弹窗

三级弹窗是居中的采集进度界面，用于显示：

- 当前缺失数据数量。
- 队列执行进度。
- 子任务名称和进度条。

它没有其他业务按钮，只有关闭叉号。根容器为：

```css
div.el-dialog
```

唯一特征是内部包含：

```css
div.dialog-title
```

### 3.4 关键 DOM 验证结果

二级弹窗打开时，一级和二级容器同时处于可见状态，而且几何信息完全一致：

```text
width: 489.4375
height: 625
x: 255.5625
y: 0
```

这证明二级弹窗完全覆盖在一级弹窗上方。

另外，三层容器内部均已验证只有一个 `i.el-icon-close` 关闭图标，因此可以在所属容器内安全定位。

## 4. 根本原因

本次问题不是单一按钮偶发失效，而是多个定位和状态判断问题叠加的结果。

### 4.1 把业务层级误当成了 DOM 组件类型

一级和二级是业务上的层级，但前端实现中它们都是 Element UI Drawer。

旧代码使用：

```python
drawer_selector = "div.el-drawer.rtl:visible"
```

等待二级弹窗时，已经打开的一级弹窗本身就满足这个选择器，所以等待会立即通过，并不能证明二级弹窗真的已经展开。

### 4.2 `:visible` 不等于可点击

`:visible` 只能证明元素有可见尺寸且没有被 `display:none` 或 `visibility:hidden` 隐藏。

它不能证明：

- 元素没有被另一层弹窗覆盖。
- 元素能够接收鼠标事件。
- 元素属于当前正在操作的业务弹窗。

因此 Playwright 可以同时报告“元素可见、启用、稳定”和“另一个子树拦截鼠标事件”。

### 4.3 `force=True` 绕过了最关键的遮挡检查

旧代码点击启动检测时使用：

```python
await page.click("#checkbutn", force=True)
```

当二级弹窗仍然覆盖在一级弹窗上时，一级 `#checkbutn` 的坐标与二级“队列任务数”控件几乎重合。

`force=True` 会绕过 Playwright 对“目标是否真正接收鼠标事件”的检查，最终导致鼠标事件命中上层的队列数量控件，表现为队列数不断减一。

### 4.4 全局叉号加 `first/last` 不能代表弹窗层级

旧代码使用：

```python
page.locator("i.el-icon-close:visible").first
page.locator("i.el-icon-close:visible").last
```

这种写法依赖当前 DOM 顺序，而不是具体弹窗的所属关系。

第一次关闭后，DOM 顺序和可见元素集合都可能变化，第二次的 `last` 不一定仍然是原先认为的二级叉号。

### 4.5 失败后只重复点击，没有改变遮挡状态

元素被覆盖时，如果重试流程只是反复点击同一个按钮，而没有关闭上层容器，那么重试不会改变页面状态，只会重复命中同一个遮挡问题。

## 5. 本次修改内容

### 5.1 使用独有子元素绑定三层容器

新代码不再把所有 Drawer 视为同一层，而是根据内部唯一锚点定位：

```python
def _primary_drawer(self, page: Page) -> Locator:
    return page.locator("div.el-drawer.rtl").filter(
        has=page.locator("#checkbutn")
    )

def _secondary_drawer(self, page: Page) -> Locator:
    return page.locator("div.el-drawer.rtl").filter(
        has=page.locator("#loseDays_shop_btn")
    )

def _progress_dialog(self, page: Page) -> Locator:
    return page.locator("div.el-dialog:visible").filter(
        has=page.locator("div.dialog-title")
    )
```

这里利用的不是 Drawer 在 DOM 中的顺序，而是容器和内部业务元素的稳定隶属关系。

### 5.2 将关闭动作限定在所属容器内

新增 `_close_layer_if_visible()`：

1. 指定层级不存在或不可见时立即返回，不等待超时。
2. 只在当前容器内查找 `i.el-icon-close`。
3. 要求关闭按钮数量必须等于一，否则明确报错。
4. 点击后等待该容器进入 hidden，不再依赖固定的 `sleep`。

`hidden` 既可兼容二级 Drawer 变成零尺寸，也可兼容某些弹窗直接从 DOM 移除。

### 5.3 区分两种页面恢复目标

#### 恢复到一级弹窗

`_restore_primary_state()` 用于日期区块之间的恢复：

```text
三级存在 -> 关闭三级 -> 等待 hidden
二级存在 -> 关闭二级 -> 等待 hidden
一级保留 -> trial 检查启动按钮可点击
```

如果只存在二级弹窗，三级检查会立即返回，不会等待三级超时。

#### 回到任务卡片大盘

`_close_all_task_layers()` 用于 Worker 初始化：

```text
关闭三级 -> 关闭二级 -> 关闭一级
```

完成后再点击指定任务卡片，进入全新的一级弹窗。

### 5.4 移除启动检测的强制点击

两处 `#checkbutn` 点击均已移除 `force=True`，改为 Playwright 普通点击：

```python
await start_btn.click(timeout=10000)
```

普通点击会检查：

- 元素唯一。
- 元素可见。
- 元素已启用。
- 元素位置稳定。
- 元素真正能够接收鼠标事件。

如果二级弹窗仍然覆盖在上方，点击会安全失败，而不是穿透层级去改变队列数量。

### 5.5 使用 `trial=True` 验证一级页面恢复

关闭二、三级后执行：

```python
await primary_drawer.locator("#checkbutn").click(
    trial=True,
    timeout=5000,
)
```

`trial=True` 会执行完整可点击性检查，但不会真正触发启动检测。

它在这里用于确认页面已经恢复到一级可操作状态。

### 5.6 所有业务按钮都限定在所属容器内

例如：

```python
start_btn = primary_drawer.locator("#checkbutn")
backfill_btn = primary_drawer.locator("span.lostDataBtn")
whole_store_btn = secondary_drawer.locator("#loseDays_shop_btn")
```

这避免了页面中隐藏的旧 DOM 或其他层级中的同类元素被误选。

### 5.7 “全店补齐”未提交时不再进入心跳判定

新流程中，只有 `click_success == True` 才会进入 120 秒心跳阶段。

提交失败时：

```text
记录提交失败
-> 精确清理二、三级弹窗
-> 跳过当前日期区块
-> 继续下一个区块
```

这既保留了 fail-forward 调度策略，也避免了“未提交 -> 无心跳 -> 误判成功”。

### 5.8 区分正常完成和卡死清理日志

`wait_for_completion_or_heartbeat()` 现在返回布尔值：

- `True`：心跳停止后重新请求后端检测，确认当前日期无缺失。
- `False`：后端仍返回缺失数据，或连续检测后结果仍不可信。

卡死区块不再记录为“成功跑完”。

### 5.9 其他日志和打包调整

- 心跳“等待出现超时”和“等待消失超时”分别记录，不再混为同一个超时原因。
- 删除每次正常心跳的高频 DEBUG 日志，避免数百条缺失数据产生无价值日志噪声。
- Worker 结尾改为“所有任务区间已遍历完毕”，不暗示每个区块全部成功。
- 打包后日志写入 exe 同目录；源码运行时日志写入脚本同目录。

## 6. 为什么修改后不会再出现同类负数问题

本次修改不能保证所有未知 UI 异常永远不发生，但已经从结构上切断了“队列任务数不断减一”这一问题的完整触发链路：

```text
二级弹窗残留
-> 一级启动按钮被覆盖
-> force=True 绕过遮挡检查
-> 鼠标事件命中二级队列数控件
-> 队列数不断减一
```

现在的防护链路是：

```text
每个区块开始
-> 精确检查三级容器
-> 精确检查二级容器
-> 如果存在则关闭并等待 hidden
-> trial 验证一级启动按钮真正可点击
-> 使用普通 click，保留 Receives Events 检查
```

即使二级容器未能成功关闭，普通点击也会因遮挡而失败并跳过当前区块，而不会穿透弹窗层级去点击队列数控件。

## 7. 本次修改未改变的业务策略

为了避免后续回看时误以为脚本需要“强一致重试”，特别记录以下设计选择：

- 区块失败后仍然可以直接跳过。
- 本次没有增加总体失败队列或任务级重试。
- 无心跳只是启动终态检查的信号，不是直接的成功/失败结论。
- 心跳停止后，清理三级、二级弹窗只是为了恢复一级操作状态；成功与否由后端缺失量复检决定。
- 不统计单次任务实际采集了多少条数据，因为当前业务更关心任务是否提交和是否完成。

## 9. 后续运行观察清单

后续回看日志时，优先关注以下信号：

1. 是否仍出现 `subtree intercepts pointer events`。
2. 如果出现，拦截元素是否仍然是队列任务数控件。
3. 是否出现“弹窗内部预期 1 个关闭按钮，实际找到 N 个”。
4. 是否频繁出现“全店补齐未成功提交”。
5. 是否出现二级弹窗无法在规定时间内进入 visible 或 hidden。
6. 心跳停止后的后端复检是否稳定，JSONL 中 `success=true` 是否与网页实际无缺失一致。
7. 日志中的“成功跑完”、“卡死并已清理”、“提交失败并跳过”是否与实际页面状态一致。

## 10. 总结

本次问题的关键认知是：

> 业务上的一级、二级弹窗，在 DOM 中可以是完全相同的前端组件类型。自动化不能只依赖类名、可见性或 DOM 顺序，而应该依赖容器与内部业务锚点的稳定关系。

修改后，脚本从“全局找可见元素并强制点击”，转变为“识别业务层级、限定操作范围、确认层级已关闭、保留 Playwright 可点击性保护”。

这不仅解决了当前队列任务数变成负数的问题，也为后续处理其他多层 UI 异常提供了可复用的框架。
