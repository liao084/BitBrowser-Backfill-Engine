# 数仓历史补采 RPA 项目思考与复盘

> 本文用于记录项目产生时的业务背景、设计思路、技术取舍和阶段性判断。它不是一份单纯的使用说明，而是一份留给未来自己的项目档案，便于在离职后、项目迭代时或面试准备中快速恢复当时的完整思路。

> 架构状态：2026-06-30 已迭代至 v2.0，共享任务池、任务账本、总体重试、页面崩溃隔离、精确弹窗分层和红色错误提示回收器均已落地。

## 一、项目背景

团队需要根据客户需求，对部分数仓任务执行历史数据补采。

过去，这项工作主要依赖人工操作：工作人员进入数仓后台，找到对应的任务卡片，手动选择补采日期，启动缺失数据检测，再点击补齐按钮完成采集。

这套人工流程可以工作，但存在明显的效率问题：

1. 工作人员不会一直守在服务器前。如果任务在采集过程中卡死，而没有被及时发现，就会长时间占用执行窗口，造成效率浪费。
2. 为了减少重复操作，人工通常倾向于设置较大的日期区块，例如 7 天、15 天甚至 30 天。但是大区块更容易触发采集异常；1 天或 3 天的小区块通常更稳定，却会显著增加人工操作次数。

因此，这个场景非常适合用 RPA 替代重复操作：机器负责小粒度、高频率地执行补采，人只负责配置任务和处理少量异常。

## 二、阶段目标与开发策略

这个项目开始于我入职后的前三周，也是团队鼓励新成员主动尝试的一项工作。

我为第一阶段设定的目标不是一次性实现一套功能完整、覆盖全部异常的调度系统，而是：

> 先构建一个能够稳定跑通主要业务流程、可以投入实际使用的框架，再根据真实运行中出现的问题逐步优化。

这一策略直接影响了后续的多个设计选择。例如，第一阶段检测到某个日期区间卡死后，不立即实现复杂的自动重试，而是清理现场、跳过当前区间并继续执行后续任务。

这样做牺牲了单个任务区间的自动恢复能力，但优先保证了整个批次不会被一个异常区间拖死。在历史补采场景下，这是对开发时间、整体吞吐量和异常处理复杂度的一次主动权衡。

目前，脚本已经投入生产使用，实际效果较好，达成了第一阶段预期目标。

## 三、技术选型

主管此前使用 Selenium 和 WebDriver 控制现有浏览器的脚本给了我启发：补采自动化不一定要重新登录系统，也可以直接接管已经登录、已经具备业务环境的浏览器。

在此基础上，我选择使用 Playwright：

- 通过比特浏览器接口获取现有浏览器的 CDP 调试地址；
- 使用 Playwright 连接并接管已登录的浏览器；
- 筛选 URL 中包含 `datatoolcenter` 的页面，将它们作为并行 Worker；
- 所有配置先拆成唯一日期任务并进入共享任务池，由可用 Worker 动态领取。

相比重新创建浏览器会话，这种方式可以复用登录状态和现有业务环境，减少登录、验证码、Cookie 管理等非核心问题。

## 四、任务配置与日期分块

脚本通过一个 `List[dict]` 形式的 `tasks_config` 描述补采任务。每条配置主要包含：

- `card`：任务卡片索引；
- `start`：补采开始日期；
- `end`：补采截止日期；
- `chunk_days`：单个任务区块包含的天数。

示例：

```python
tasks_config = [
    {
        "card": 5,
        "start": "2025-09-01",
        "end": "2025-09-30",
        "chunk_days": 1,
    }
]
```

引擎会把一个大日期范围拆成多个小区间，再依次执行。例如，9 月 1 日至 9 月 30 日使用 `chunk_days=3` 时，会生成 10 个连续的三天任务区块。

这个抽象解决了人工操作中的核心矛盾：既可以使用更稳定的小区块采集，又不会因为区块变小而增加人工工作量。

## 五、整体架构

当前脚本可以理解为三个层次：

### 1. 配置层

`tasks_config` 描述要处理的任务卡片、日期范围和分块大小，使业务任务与执行代码分离。

### 2. Worker 执行层

每个符合条件的 `datatoolcenter` 标签页作为一个 Worker。引擎先把全部配置拆成唯一日期任务，再放入共享的 `asyncio.Queue`。所有健康 Worker 动态领取任务，不再把某一段配置固定绑定到某个标签页，从而减少不同卡片和日期区间执行时长差异造成的空闲等待。

Worker 负责：

1. 打开指定任务卡片；
2. 输入当前日期区间；
3. 启动缺失数据检测；
4. 判断是否存在缺失数据；
5. 打开补齐弹窗并启动全店补齐；
6. 监听心跳并判断当前区间是完成还是卡死；
7. 清理异常弹窗并继续下一个日期区间。

每次任务尝试的最终结果都会由带异步锁的 JSONL 任务账本串行记录。首次执行结束后，引擎从账本读取失败项，统一进入唯一一次总体重试；页面崩溃、关闭或浏览器断连时，对应 Worker 会停止领取新任务，剩余健康 Worker 继续消费共享队列。

### 3. Context 级 GC 层

京东商智页面并不直接作为 Worker，而是由数仓后台在实际采集时动态创建。引擎在 Browser Context 层监听新页面，为商智页面创建独立的生命周期监控任务。

GC 与具体 Worker 解耦，只关心商智页面是否仍有心跳；页面超过规定时间没有心跳，就被认为是遗留的僵尸执行页并关闭。

## 六、心跳机制与卡死判断

### 1. 为什么选择“同步成功”弹窗作为心跳

数仓后台和京东商智执行页在成功采集数据时，都会持续出现“同步成功”弹窗。通常每采集一条数据就出现一次，间隔约 3～4 秒，弹窗本身也会在约 3～4 秒后消失；繁忙时页面上可能同时存在两个弹窗。

因此，这个弹窗可以作为采集过程仍在推进的可观测信号。只要弹窗持续出现，就说明任务仍然活跃；长时间没有新弹窗，则说明任务已经完成或者发生卡死。

### 2. `datatoolcenter` Worker 的 120 秒判断

Worker 在 120 秒内没有收到新心跳后，并不会只凭静默直接认定任务成功或失败，而是执行后端终态复检：

- 按层级关闭三级、二级弹窗并恢复一级任务弹窗；
- 重新注入当前日期，点击“启动检测”请求后端状态；
- 后端确认无缺失时才记录成功；仍有缺失或结果不可信时记录失败。

因此，120 秒静默本质上只是一次“状态转换触发器”。三级弹窗不再作为成功证据，最终以重新检测得到的缺失状态为准。

### 3. 京东商智页面的 180 秒判断

商智页面由独立 GC 监控。每次捕获到“同步成功”心跳，相当于刷新页面的生存期；如果 180 秒没有新心跳，GC 就将该页面视为僵尸页面并关闭。

Worker 使用 120 秒，商智 GC 使用 180 秒，中间错开 60 秒，目的是让数仓后台先完成业务状态判断和弹窗清理，再由 Context 级 GC 处理未正常关闭的执行页面，降低两个清理机制互相干扰的概率。

## 七、商智页面捕获方案的演进

最初的设想是：在数仓页面点击“全店补齐”时，使用类似 `expect_popup` / `expect_page` 的方式捕获这次操作打开的商智页面，并将它与当前 Worker 建立对应关系。如果 Worker 卡死，就关闭与它关联的商智页面。

后续通过实际观察发现，这种绑定关系并不稳定：

1. 一个任务队列中可能包含多条缺失数据；
2. 商智页面完成其中一段实际采集后会自动关闭；
3. 数仓后台随后可能重新打开一个新的商智页面继续采集；
4. 最初捕获到的页面消失后，Worker 与后续新页面的关联也随之丢失。

因此，最终方案不再强行维护 Worker 与商智页面的一一对应关系，而是在 Browser Context 层统一捕获所有新页面，并为符合条件的商智页面启动独立 GC。

这种设计把两个问题拆开了：

- Worker 只负责数仓后台的任务调度和业务状态；
- GC 只负责商智执行页的存活检测和资源回收。

商智页面不需要知道自己属于哪个 Worker，GC 也不参与 Worker 的任务判断。这种解耦降低了动态页面生命周期带来的关联复杂度。

## 八、第一阶段的工程取舍

### 1. 卡死后跳过，而不是立即自动重试

完整的重试机制需要考虑很多问题：

- 是否立即重试；
- 最多重试多少次；
- 是否需要更换 Worker；
- 重试前如何确认旧商智页面已经清理；
- 如何避免同一批数据被重复下发；
- 连续失败后如何记录和恢复。

这些问题会显著增加第一阶段的开发和验证时间。因此第一阶段的选择是：识别异常、清理弹窗、记录日志、跳过当前区间并继续运行。

这并不保证每个区间都自动成功，但可以避免一个异常任务阻塞整个批次，是为了尽快交付可用系统而做出的阶段性选择。

在第一阶段框架经生产验证后，v2.0 增加了任务账本和一次总体重试。这个演进没有推翻原来的 fail-forward 原则：单次失败仍不会阻塞队列，只是在全部首次任务完成后，把账本中明确失败的任务再执行一次，形成可控而有限的恢复闭环。

### 2. 依赖业务页面信号，而不是追求绝对状态

由于无法直接获得数仓后台和京东商智内部任务状态，脚本只能根据弹窗、弹窗层级和页面生命周期推断执行情况。这是一种带有业务经验的启发式判断。

它并非理论上绝对准确，但目前已经通过多次实际运行验证，能够覆盖主要生产场景。后续优化应优先由真实故障驱动，而不是为了形式上的完整性过早增加复杂度。

## 十一、可用于面试回顾的项目主线

面试时可以围绕以下主线介绍：

> 我在入职后发现历史补采依赖大量人工操作。为了减少操作次数，人工倾向于使用大日期区块，但大区块又更容易触发采集异常；使用小区块虽然稳定，却会显著增加人力成本。因此我用 Playwright 接管已登录浏览器，把数仓页面抽象为并发 Worker，并通过配置控制任务卡片、日期范围和分块大小。针对页面可能卡死的问题，我根据“同步成功”弹窗设计了两级心跳：Worker 在 120 秒静默后恢复一级弹窗并重新请求后端缺失量，只有复检无缺失才成功；Browser Context 层则用独立 GC 在 180 秒后回收业务僵尸页面。第一阶段优先建立可投入生产的 fail-forward 框架；经过真实运行验证后，我又把固定任务分配升级为共享任务池，加入 JSONL 任务账本、总体重试、Daily 单任务即时回队、异常 Worker 隔离和事件驱动的错误提示回收，使局部故障不再放大为整个批次停滞。

## 十二、已解决问题：多个心跳弹窗与 `detached` 等待

早期心跳监听在捕获到“同步成功”弹窗后，会用同一个选择器等待所有匹配元素进入 `detached` 状态，再开始等待下一次心跳。

正常情况下，弹窗约 3～4 秒消失。最初把具体节点的消失等待设为 15 秒；后续多脚本同时启动产生资源高峰时，15 秒仍可能不足，因此当前版本已经放宽为 30 秒。

v2.0 已将该逻辑改为：先保存本次捕获到的具体 `ElementHandle`，之后只等待这个节点进入 `hidden`。后续新出现的成功提示不会替换当前等待目标，也不会阻止当前节点完成生命周期判断。

同时，“等待新心跳出现”和“等待当前心跳节点隐藏”已经拆成不同异常分支，日志能够区分 120/180 秒无新心跳与单个提示节点 30 秒未隐藏。

这个问题的解决过程沉淀出一条可复用原则：

1. 同类元素可能并发出现时，不应使用全局选择器代表某一个节点的生命周期；
2. 捕获事件后应固定具体节点，再观察该节点隐藏或移除；
3. 不同语义的超时应拆分记录，避免一个异常分支混合多个业务状态。

当前状态：已解决并通过隔离页面测试。

## 十三、红色错误提示堆积与事件驱动回收

生产运行中发现，当商智登录失效或服务器存储接口异常时，数仓 Worker 页面会为每条失败数据生成一个不会自动消失的红色错误提示。提示长期累积后会覆盖页面并拦截鼠标事件，使 Playwright 即使定位到了正确按钮也无法安全点击。

最终没有恢复 JS 或 `force=True` 强制点击，因为精准 Locator 只能证明“找对了元素”，Playwright 的 actionability 检查仍负责证明“此刻适合操作”。绕过遮挡会掩盖页面异常，也无法解决错误节点继续堆积的问题。

新增的错误提示回收器为每个 Worker 页面启动独立后台协程，并使用 `wait_for_selector(timeout=0)` 事件等待新的未标记错误节点。空闲时协程挂起，不进行每秒 DOM 扫描；捕获后记录错误内容、固定具体节点、保留 30 秒供人工观察，再通过该节点内部的专属关闭按钮执行 Playwright 安全点击。

这套设计与商智 GC 使用相同的守护思想，但作用层级不同：GC 回收超时的整个商智页面，错误提示回收器只处理 Worker 页面内部过期的 UI 节点。

### 生产案例：DOM 中的“幽灵红色提示”

后续一次生产运行暴露出更隐蔽的情况：红色错误提示节点仍存在于 DOM 中，监控器能够识别它，但人工在页面上看不到该提示，Playwright 的常规点击也始终无法通过可见性或遮挡检查。旧实现因此会在宽限期结束后反复尝试同一个节点，约每 30 秒产生一次超时日志。这个异常没有中断主业务，却造成了持续的日志噪声。

最终处理分为两层：首先，节点一经捕获就永久保留“已调度”标记，即使关闭失败也不再重复调度同一个节点；其次，只对已经精确绑定在该错误提示内部的关闭叉号增加 `node.click()` DOM 降级。常规点击仍是首选，只有命中测试因遮挡失败时才执行精准 DOM 点击，业务按钮则绝不绕过 Playwright 的安全检查。

这个案例让我意识到，后台守护机制本身也要满足幂等和故障隔离：监控器即使无法修复目标，也不应该把一个静态异常放大为无限重试和日志洪泛。同时，DOM 精准点击并不是通用的“强制点击”，它只适合语义明确、作用可逆、已被容器范围严格限定的关闭控件。

## 十四、生产案例：页面失去响应与 Worker 熔断

一次通宵运行从前一晚约 22 点稳定持续到次日 6 点。随后数据中心页面出现人工点击也没有反应的前端冻结：页面仍在浏览器中，CDP 连接也没有立即断开，因此它不是传统意义上的“页面已关闭”。其中一个 Worker 在初始化阶段连续失败，却继续从共享任务池领取任务，最终产生 174 次初始化失败；另外四个 Worker 则停在三级弹窗的 `count()` / `is_visible()` 查询上。由于 `asyncio.gather()` 一直等待这些协程，首轮任务无法收尾，也无法进入总体重试和结果汇总。

这次故障揭示了共享任务池架构中的一个关键边界：任务失败和执行者失败必须分开处理。业务区间失败可以记录后继续前进，但 Worker 自身失去操作能力后必须及时隔离，否则它会快速消耗共享队列，把页面故障放大成大量任务失败。

对应的改进包括：

1. 为本应瞬间完成的 DOM 数量、可见性和页面健康查询增加 `asyncio.wait_for()` 外层硬超时，但不给可能持续数小时的完整补采任务设置总超时；
2. 使用 `document.readyState` 做一次受限时保护的 JavaScript 往返。这里主要验证渲染进程能否及时执行代码并返回，而不是只接受某个加载阶段；
3. 精确关闭三级、二级弹窗时，常规点击失败后仅对其内部唯一叉号使用 DOM 点击，并验证弹窗确实隐藏、页面仍能响应；
4. 普通任务页面连续初始化失败 3 次后触发 Worker 熔断，停止从共享池领取新任务；页面无响应、崩溃或断连则立即熔断；
5. `asyncio.gather(..., return_exceptions=True)` 隔离单个 Worker 协程的意外退出，使其他 Worker 和本轮收尾逻辑仍能继续执行。

这次迭代把系统从“任务级 fail-forward”推进到了“任务与执行者双层容错”。面试中可以用它说明：日志分析不只用于修一个超时报错，还能反推出并发调度系统缺少的失效边界，并据此补上健康探测、熔断、隔离和收敛机制。
